Scope · Next.js
The thesis
Infinite scope is not a scope. If we don't control the website runtime, we can't control the experience.
decided · post-LLM research

"Just raise a PR" looks scoped, but it creates infinite scope — the final outcome depends on each customer's CI/CD, env vars, build setup, preview deployments, hosting, deploy speed, and repo conventions. That gives Kite low control but high accountability: even when Kite writes the right code, the experience still breaks if the build fails, the preview doesn't exist, or env vars are missing. From the user's point of view, it's still a Kite failure. So Kite should not be just a PR generator. Kite should import the website into a Kite-controlled runtime where we can understand, modify, preview, approve, and deploy. GitHub stays connected for source control and sync, but the core website experience is owned by Kite.

The outcome we're designing for
north-star quote

Every decision below should be evaluated against one sentence — the moment we want the user to feel the first time they ask Kite to change their site:

"I asked for a change, Kite understood it, previewed it, and shipped it almost instantly."

That is not possible if every change has to wait on the customer's CI/CD and deployment pipeline. The snap-speed wow factor requires that Kite own the runtime end-to-end — at least for the categories of website we choose to support.

User stories · jobs to be done
4 primary personas

The user looking to use Kite AI CMO is, in almost every case, a marketer or a founder acting like one — someone whose job is to ship copy, pages, and campaigns, not to debug builds. The CMO analogy is load-bearing: hiring a CMO doesn't mean rebuilding the business infrastructure. A real CMO first understands the existing business, then adds the right capabilities. Kite should behave the same way.

Create · greenfield

The founder shipping a marketing site from scratch

"I have a product and a pitch. I don't have a website yet. I want one live by tonight, owned by me on my GitHub, but I don't want to think about hosting, previews, or CI/CD."

Job to be done When I have a brand and a few pages in my head, help me get a production-quality marketing site live in one session, on infra I can take with me, without ever opening a terminal.

Improve · existing repo

The marketer iterating on a live Next.js site

"The site already exists, the dev team built it. I want to change the hero headline, swap a CTA, add a pricing page — and see the change live, not wait two days for the next release."

Job to be done When the website is already in a GitHub repo, let me import it, understand what's there, propose changes, preview them on a Kite-controlled URL, and ship to production when I approve.

Scale · ongoing optimisation

The growth lead running constant experiments

"I'm not making one change. I'm making twenty a week. Different headlines on different pages, an A/B on the pricing block, a new testimonial section by Friday."

Job to be done When I'm in a rhythm of weekly experiments, make each change feel like one prompt and one approval — and make Kite remember the brand voice, components, and conventions so I'm not re-explaining context every time.

Repair · diagnose & fix

The owner whose site is drifting

"My site loads slowly on mobile, the OG tags are wrong, and half the blog posts have broken images. I don't even know where to start."

Job to be done When I describe a problem in plain English, audit the site, surface what's actually wrong, propose specific fixes with a preview, and let me ship them without becoming a developer.

The two operating scenarios
create · improve

Underneath the four user stories, Kite has to operate in two technical modes. For a great UX, Kite needs to be a strong GitHub App in both — creating new projects and improving existing ones — with direct access to the codebase.

Scenario A · primary v1

Improve an existing GitHub repo

Bring your site → Kite owns the runtime

User installs the Kite GitHub App on their repo. Kite imports the website into a Kite-controlled runtime, understands the stack, and from then on every change runs through Kite's loop. GitHub stays connected for source control and sync.

  • Detect the stack first — only ask for GitHub when we can credibly manage the tech.
  • Pre-analysis → analysis report → credit card → connect GitHub → in-depth verification → charge.
  • Kite owns build, preview, deploy. The customer's CI/CD doesn't gate the wow moment.
Scenario B · greenfield

Create a new website end-to-end

Start in Kite → land in their GitHub

User starts with a prompt: "Here's my business, here's what I need." Kite generates the site inside its own runtime, then provisions a fresh GitHub repo via the GitHub App so the user owns the code from day one.

  • No infra decisions for the user. Hosting, previews, CI/CD all live inside Kite by default.
  • GitHub is the export layer — the user can leave with their code anytime.
  • Same loop as Scenario A. Same product, not a separate flow.
The Kite loop · how every change moves
5 stages

Once the website is imported, every change — Scenario A or B — runs through the same five steps. This is the loop the user feels.

01

Understand

Read the repo, the stack, the brand, the existing components. Form a working model of the site before proposing anything.

02

Modify

Make the change the user asked for, in the right files, following the project's existing conventions — not Kite's.

03

Preview

Render the change on a Kite-controlled preview URL within seconds. No waiting on the customer's pipeline.

04

Approve

The user reviews the preview and either approves, asks for revisions, or rolls back — all inside Kite.

05

Deploy

On approval, Kite ships to production and syncs the commit back to the user's GitHub main.

Architectural considerations
resolved · open

Three concerns surfaced repeatedly during the debate. Two are now constraints we will design around; one is a pattern we want to learn from.

DP ⇄ prod parity
Risk: if Kite runs DPs in a Kite environment but production lives on the customer's existing setup, a change that works in the DP can still break in production. Implication: for v1, the DP environment and the production environment must be identical — both Kite-controlled — or the wow moment is at the mercy of an environment we don't own.
"Kite branch" model
Rejected. If we treat a kite branch as the main branch inside Kite, the user has to "publish" twice — once from the DP into the kite branch, again from kite into main. That's two mental steps to do one thing. Bad UX. We will instead branch off main per change and merge directly into main on approval.
Vercel GitHub App
Pattern to study. When the Vercel GitHub App is installed, it starts producing DPs for every push and exposes a production URL that may or may not be the customer's actual production. The split between "Vercel's preview/prod" and "the customer's real prod" is the same shape as our problem — and worth a focused teardown before we lock our own architecture.
v1 scope · what we are explicitly not building
bounded

Infinite scope killed the PR-only approach. We should not let it kill the runtime approach either. v1 is deliberately narrow.

In scope · v1

Next.js marketing sites

one stack · run it really well
  • Next.js as the only supported framework.
  • Marketing-style websites — landing pages, pricing, blog, simple CMS.
  • GitHub App as the connector. No GitLab, no Bitbucket.
  • Kite-owned runtime for build, preview, and deploy.
  • Both scenarios (import existing, create new) on the same loop.
Not in v1

Anything that fragments the runtime

say no — for now
  • Other frameworks (Astro, Remix, plain HTML, WordPress).
  • Multi-repo changes within a single Kite thread.
  • Customer-owned CI/CD as the deploy path.
  • Web apps with auth, dashboards, server-side state.
  • "Just raise a PR" as a fallback mode — it dilutes the loop.
POC · available today appsmithorg / kiteport

Experiment with kiteAI as a Next.js repository

The fastest way to pressure-test the v1 scope is to run our own marketing site through it. The kiteAI website is already published as a Next.js repository under the kiteport monorepo — use it as the first import target, the first DP, the first end-to-end loop. If the loop is great here, the loop is great.

Open questions for the team
need a decision
Q1

How permissive should the GitHub App permissions be on day one?

GitHub Apps can deploy, make CI/CD changes, and modify code. We probably want all three for the loop to work — but we should be deliberate about the install-time consent screen and what we promise not to touch (e.g. unrelated branches, secrets).

Q2

If a user later wants to leave Kite, what does a clean exit look like?

GitHub is the export surface, but if Kite has been the runtime, the user is also leaving the build/preview/deploy layer. We should know our answer before we onboard the first paying customer, not after.

Q3

How do we handle a change that genuinely spans multiple repos?

v1 says no, but real marketing sites sometimes touch a content repo and a code repo. Worth knowing whether we defer this to v2 or whether the loop should accept it from the start, even if the UX is rougher.

Q4

Where does the credit-card step actually go in the funnel?

Current draft: pre-analysis → analysis report → credit card → connect GitHub → in-depth verification → charge. We should validate that "ask for the card before the GitHub install" doesn't kill conversion for the marketer persona.
