The thesis
Every one of these users hires Kite to finish the job, not to hand them a pull request. If Kite doesn't own the runtime — build, preview, deploy — then we don't own the outcome. Infinite scope is not a scope.

01
Create · greenfield No existing repo First-time user

"I have a pitch deck and a demo on Monday. I don't have a website."

Maya, solo founder. Two months into a stealth AI tax-prep tool called Lumenfile. She can write copy and pick a font; she has never touched a CI pipeline. She has a GitHub account she made for an old hackathon.

Hires Kite as
The brand-launch CMO
The moment

Friday · 8:42 pm The pitch deck has gotten three "where's your site?" emails this week. She opens Kite on a laptop with one tab and a logo SVG on her desktop. There is nothing else. She has until Sunday night before the Monday demo.

What she says to Kite

"Build me a five-page marketing site for Lumenfile — landing, pricing, a short blog, an about, and a demo CTA. AI tax prep for indie consultants. Calm, serious, not playful."

Job to be done
When I have a product and a brand in my head but nothing built, get me a production-quality marketing site live in one session — on infra I'll own through my own GitHub the moment I want to walk away — without me ever opening a terminal or making a Vercel account.
✓We win when

A live, brand-coherent site at a real URL within 30 minutes. A Kite GitHub App has provisioned lumenfile/site on her account. She edits the headline by speaking to Kite, sees it on a preview URL in seconds, and approves. The Monday demo opens with a working website.

!We lose when

Kite generates a Next.js zip, hands her a guide to Vercel, and disappears. Or: the site is live but lives only inside Kite with no GitHub export, so she's locked in on day one. Either failure costs the trust the demo was supposed to build.

02
Improve · existing repo Next.js · App Router Has a dev team

"The site exists. I just want to change the hero headline today, not next sprint."

Devan, head of marketing at a 40-person Series A company. The marketing site is a Next.js repo the engineering team built nine months ago. Every copy change goes into a Linear ticket and ships "with the next release." That is not fast enough.

Hires Kite as
The on-staff site editor
The moment

Tuesday · 11:15 am The CEO just pasted a new positioning line into Slack and said "let's try this live before the all-hands." Devan has historically had to write a ticket, wait two days, and watch the engineer get distracted by an incident. He installs the Kite GitHub App on the marketing-site repo.

What he says to Kite

"Replace the homepage H1 with this: 'Audit-ready books, without the auditors.' Keep the existing eyebrow. Push it to a preview I can show the CEO before lunch."

Job to be done
When the website is already a real codebase someone else wrote, import it, understand its components and conventions, make the change in the right file, preview it on a Kite URL within seconds, and merge to main only when I approve — without me having to learn the repo or wait for the next release train.
✓We win when

Kite recognises this as a Next.js app, finds the <Hero/> component, makes the smallest possible change, and produces a preview URL Devan can paste into Slack in under five minutes. The PR back to main looks like one Devan's engineers would have written themselves.

!We lose when

Kite rewrites the hero component "to be cleaner," touches three other files, and the engineering team rejects the PR. Or: Kite raises the PR but the preview depends on the company's own CI/CD, which is slow or flaky — so Devan still can't show the CEO anything today.

03
Scale · ongoing experiments Weekly cadence Brand is fixed

"I'm not making one change. I'm making twenty a week, and Kite should remember every one of them."

Priya, growth lead at a B2B SaaS. She runs a permanent backlog of pricing-page tweaks, headline A/Bs, new testimonial blocks, and seasonal banners. Her job is throughput — turning ten experiment ideas into ten shipped variants.

Hires Kite as
The experimentation hand
The moment

Every Monday · 9 am Priya pastes the week's experiment list into Kite — five headline tests, a new logo strip, a swap of the pricing FAQ. She does not want to re-explain the brand voice, the components, or what "do not touch" means. Kite should already know.

What she says to Kite

"Spin up A/Bs for these five hero headlines. Use the same testimonial block, no new components. Ship them as one branch, not five PRs. Tell me which ones are live by Friday."

Job to be done
When I'm running a constant rhythm of small experiments, make each change one prompt and one approval, batch them sensibly, and persist context across weeks — brand voice, component library, "don't touch" zones — so I'm a growth lead, not a project manager for Kite.
✓We win when

Five variants live behind a feature flag by Wednesday, all using the existing testimonial component, all on previews she can share with the CRO. Kite has remembered last month's brand-voice corrections and applied them silently. Throughput goes up; her judgment is what gets scaled, not her time.

!We lose when

Kite treats each ask like the first one. It re-asks for brand guidelines, creates five near-duplicate components, opens five separate PRs Priya has to babysit, and forgets the "don't touch the pricing JSON" rule she explained last week. Cognitive load shifts from engineering to her.

04
Repair · diagnose & fix Inherited site Non-technical owner

"My site loads slowly, OG tags are wrong, half the blog images are broken. I don't know where to start."

Hari, founder of a five-year-old bootstrapped product. The original developer left two years ago. The site mostly works. It also mostly doesn't. He has the GitHub login somewhere; he hasn't opened the repo in a year.

Hires Kite as
The diagnostic technician
The moment

Sunday · 10 pm A friend tells Hari his site is "really slow on mobile" and the LinkedIn preview shows the wrong image. He doesn't know what an OG tag is. He doesn't want to learn. He just wants someone to fix it before Monday morning.

What he says to Kite

"Something's wrong with my website. It loads slow on phones and looks broken when I share it on LinkedIn. Find what's wrong and fix it. Don't make me read anything."

Job to be done
When I describe a problem in plain English, audit the site, surface what is actually wrong in one human-readable list, propose specific fixes with previews, and let me ship them with one tap — without ever making me become a developer to own the site I already own.
✓We win when

Kite returns a ranked list — "1. Hero image is 4.2 MB, served unoptimised. 2. OG tag points to a deleted asset. 3. Three blog posts reference missing CDN images." Each has a preview of the fix and a single approve button. Hari ships all three in ten minutes.

!We lose when

Kite produces a 40-item Lighthouse report and tells him to "consider adding next/image." Or: Kite proposes fixes but routes them through his abandoned CI pipeline, which has been broken for eight months. Nothing ships, and Hari's instinct that "I can't fix it myself" is confirmed.

Section · 02 of 03

One loop, shared by all four stories.

Every story above lives or dies on the same five-step loop. The names above the dots are not abstractions — they're the literal moments Maya, Devan, Priya, and Hari experience.

5 steps
Kite-owned runtime
01

Understand

Read the repo or the prompt. Form a working model of the site, its stack, its brand, and its conventions.

02

Modify

Make the smallest correct change in the right files, in the project's existing style — not Kite's.

03

Preview

Render the change on a Kite-controlled URL in seconds. The customer's CI/CD is not in the critical path.

04

Approve

The user looks at it. They approve, ask for a revision, or roll back — all inside Kite, in plain language.

05

Deploy

On approval, Kite ships to production and syncs the commit back to main on the user's GitHub.

Section · 03 of 03

What the stories force us to build.

Working backwards from those four moments, three things become non-negotiable, and one set of things has to be deferred.

scenarios · scope · risks
Two scenarios, one product
Scenario A · primary

Improve an existing GitHub repo

Devan, Priya, Hari live here
  • Detect the stack first. Only ask for the GitHub install once we know we can credibly run their site.
  • Funnel: pre-analysis → analysis report → credit card → connect GitHub → in-depth verification → charge.
  • Kite owns build, preview, deploy. The customer's CI/CD is not in the wow path.
  • Branch off main per change. Merge to main on approval. No "kite branch."
Scenario B · greenfield

Create a new website end-to-end

Maya starts here
  • Begins with a prompt and a brand brief. Kite generates inside its own runtime.
  • Lands in their GitHub via the Kite App the moment they want to own the code.
  • No infra decisions for the user. Hosting, previews, CI/CD all live inside Kite by default.
  • Same loop as Scenario A. Same product surface, not a separate flow.
Architectural calls already made
DP ⇄ prod parity
If Kite runs DPs in a Kite environment but production lives on the customer's existing setup, a change that works in the DP can still break in production. For v1, the DP and production environments must be identical — both Kite-controlled. Otherwise the wow moment is rented from infra we don't own.
"Kite branch" model
Rejected. Treating a kite branch as Kite's main means the user has to publish twice — once from the DP into kite, again from kite into main. Two mental steps to do one thing. We branch off main per change and merge directly into main on approval.
Vercel GitHub App
Pattern to study, before we lock our own. Installing the Vercel App starts DPs for every push and exposes a production URL that may or may not be the customer's real prod. The split between "Vercel's prod" and "the customer's prod" is the exact shape of our problem. Worth a focused teardown.
v1 scope · narrow on purpose
In scope

Next.js marketing sites, run really well

one stack · two scenarios · one loop
  • Next.js as the only supported framework.
  • Marketing-shape sites — landing, pricing, blog, simple CMS.
  • GitHub App as the only connector.
  • Kite-owned runtime for build, preview, deploy.
  • All four user stories on the same five-step loop.
Explicitly not v1

Anything that fragments the runtime

say no, on purpose, for now
  • Astro, Remix, plain HTML, WordPress.
  • Multi-repo changes inside one Kite thread.
  • Customer-owned CI/CD as the deploy path.
  • Apps with auth, dashboards, server-side state.
  • "Just raise a PR" as a fallback. It dilutes the loop.
POC · the obvious first target

Run our own marketing site through the loop first.

The kiteAI website is already a Next.js repo inside the kiteport monorepo. It is the cheapest, sharpest pressure-test of the v1 scope we have. If we can't run the loop on our own site, we can't run it on anyone else's. If we can — every demo writes itself.

Still open · need the team's call
Q1

How permissive should the GitHub App be on install?

The loop needs deploy + CI/CD + code. We want all three. But the install-time consent screen is also the first impression — we should be deliberate about what we promise not to touch (unrelated branches, secrets, other repos).

Q2

What does a clean exit from Kite look like?

GitHub is the export surface, but if Kite has been the runtime, the user is also leaving build/preview/deploy. We should know our answer before the first paying customer, not after the first churn email.

Q3

Where does the credit card actually sit in the funnel?

Current draft: pre-analysis → report → card → GitHub → verification → charge. We should validate that asking for the card before the GitHub install doesn't kill conversion for the marketer persona — Devan, specifically.

Q4

When a real change spans multiple repos, do we defer or absorb?

v1 says one repo per thread. Real marketing sites sometimes touch a content repo and a code repo. Worth a position before Priya hits it in week three.
