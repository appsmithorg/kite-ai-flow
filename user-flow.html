Step-by-step

Why they drop, ranked. And what we'd do about it.

Three reasons per step, ordered by how often we expect them. The right column is the cheapest mitigation that doesn't break the rest of the funnel. Numbers are pre-data estimates — replace them with real values after the first 100 sign-ups.

6 steps
18 reasons
12 mitigations
A

Pre-analysis · "Paste your URL"

User lands on kite.ai, pastes a URL. Kite fetches the site, identifies the stack, decides whether it can credibly manage it.

100%
Entering
−38% to next
!Why they drop
A1
Unsupported stack. The URL is a WordPress, Wix, Squarespace, or Webflow site. Kite returns "we can't manage this yet." ~22% of pastes
A2
Auth-walled or staged URL. The site is behind a login or only exists on a staging domain Kite can't reach. ~11% of pastes
A3
Curiosity-only. User pasted to see what Kite does, never had intent to act. Bounces before the analysis finishes. ~5% of pastes
✓What we do about it
A
"Not yet" with a waitlist. For unsupported stacks, capture email + framework. Don't end the journey — convert the no into a signal for what to support next.
B
Manual URL override. Let the user paste a public preview URL, a Vercel deployment, or a GitHub raw link — anything Kite can crawl.
C
Streaming analysis. Show partial results before the full scan completes — keeps curious pasters from bouncing.
B

Analysis report · "Here's what we'd do"

Kite shows what it found, previews one improvement on a specific section of the site, and asks the user to sign up to see the rest.

62%
Surviving
−34% to next
!Why they drop
B1
Findings feel obvious. Kite returns generic suggestions ("optimise images, add OG tags") the user has seen in every Lighthouse report. No wow moment. ~18%
B2
Findings feel wrong. Kite misreads the site (calls a B2B SaaS landing a "personal portfolio"), loses credibility before the sign-up button. ~10%
B3
"I'll come back." User is impressed but not in build-mode right now. Closes the tab intending to return. Doesn't. ~6%
✓What we do about it
A
One specific, non-obvious finding. Surface something the user couldn't have found with a free tool — a competitor pattern, a copywriting weakness, a structural opportunity.
B
Live preview of one fix. Don't tell — show. Render the proposed change on a Kite preview URL the user can click into immediately, before any sign-up.
C
Magic-link return path. Email the report + preview link to the user. If they don't sign up, they still have a way back in.
C

Sign up · email & verify

Free account, no card. The trial wedge starts here — but only commits when the GitHub App is installed in step D.

41%
Surviving
−37% to next
!Why they drop
C1
Wrong person at the keyboard. A marketer signs up but realises they can't actually install a GitHub App on the site repo without VP-Eng approval. Stalls. ~22%
C2
"Why do you need my email?" Friction at the email-verification step from users who already saw what Kite can do and feel the gate is unjustified. ~9%
C3
Personal vs work email. User signs up with personal email, then can't connect a GitHub account tied to their work org. Loops back, gives up. ~6%
✓What we do about it
A
"Loop in your engineer" path. Mid-flow, let the marketer email a one-click magic link to their VP-Eng with the install pre-approval flow on the other end.
B
Google / GitHub SSO first. Skip email-verify for users who can SSO. Reduces the C2 friction without breaking the C3 case.
C
Account-mismatch detection. If the user signs up with personal email but the URL belongs to a work org, surface the mismatch before they get to D.
D

GitHub App install · trial begins

User installs the Kite App on the site repo. Trial runs in PR-only mode — no infra moves, easy exit. The first 14 days.

26%
Surviving
−35% to next
!Why they drop
D1
Platform lead refuses install. VP-Eng won't approve a third-party GitHub App with deploy-adjacent permissions on the production repo. ~20% — the load-bearing risk from v0.4
D2
Permission scope feels too broad. Marketer sees the consent screen, doesn't understand half the scopes, declines on instinct. ~9%
D3
"I just wanted to look." The install is a real commitment. Users who hadn't decided to act bail right here. ~6%
✓What we do about it
A
Minimum-viable permission set. Trial mode only needs contents:read + pull_requests:write. Show that. Defer the deploy scopes until step F.
B
Plain-English scope explainer. Inline annotations on the consent screen — "this lets Kite read your code" / "this lets Kite open PRs" — no GitHub jargon.
C
"Try on a sample repo" escape hatch. Let hesitant users see the PR-only loop on a public sample repo before pointing it at their own.
E

First PR shipped · loop completes once

User asks Kite for a change. Kite proposes it. Their team reviews and merges. The PR-only loop has run end-to-end for the first time.

17%
Surviving
−47% to next
!Why they drop
E1
Engineers reject the PR. Kite's diff isn't in the team's house style — wrong import ordering, wrong component pattern, touched files it shouldn't have. PR sits, then closes. ~5%
E2
No change worth asking for. Trial expires before the user has a real ask. They tried Kite without a job to give it. ~3%
E3
Repo too complex. Custom monorepo, exotic Webpack config, half-finished migration. Kite can't form a coherent plan. ~1%
✓What we do about it
A
"Style match" pass. Before opening any PR, run a style/convention pass against the repo's recent merged commits. Match — don't impose.
B
Trial activation nudge. If 7 days pass without a single ask, surface 3 specific, repo-aware suggestions Kite can ship immediately.
C
Complexity gate at D, not E. Catch unmanageable repos at install time and warn the user — better than failing silently in step E.
F

Charge · full takeover activation

Card on file, in-depth verification, scope-expansion consent. The user graduates from PR-only to Kite-owned runtime — and we charge.

9%
Surviving
Paid customer
!Why they drop
F1
"PR-only is enough." The trial was useful, but the snap-speed wow isn't worth the lock-in step. They stay on free PR-only forever — or churn. ~4%
F2
Verification finds blockers. Kite discovers "extras" (custom auth, CMS hooks, A/B service, edge configs) it can't replicate. The full-takeover promise breaks. ~3%
F3
Price shock. Trial was free; full takeover pricing is the first dollar amount the user sees, and it doesn't match the value they perceive. ~1%
✓What we do about it
A
Make the snap-speed visible during trial. Show a "what changes when you upgrade" comparison inside trial — same change, with and without Kite-owned deploy. Time saved in seconds, not slogans.
B
Pre-flight verification before billing. Run the "can we actually own this runtime" check as the first step of conversion, not the last. Refund-protect users whose extras can't be replicated.
C
Show price after value is felt. Surface pricing tied to the actual changes shipped during trial ("you saved 6 engineering hours · cost would have been $X") — not as an abstract plan grid.
