The flow · two paths · one decision that splits them

One entry. Two paths.
Eight stations. Six exits.

A customer opens Kite and asks for a change. Everything that follows runs on one of two lines: the Gold line (move to Kite, full takeover) or the Blue line (PR-only, stay on your own infra). The two share the first five stations, then split at the big decision, then diverge to two different terminal stops. Every red exit marks a place where customers leave.

Gold line · full takeover

"I need it snappy. Move my site to Kite."

Kite replicates the codebase, owns build / preview / deploy on a separate URL per environment. GitHub stays the source of truth. Best when speed matters more than reversibility.

Blue line · PR-only

"I just need a single page. Raise the PR."

Kite generates the change, opens a PR against the customer's repo, and their existing CI/CD ships it. Best when the change is small or the customer doesn't want anything moving infra-side.

★ The crux
Every customer hits "Decision 01" (have a repo) — easy. They all hit "The Big Decision" — hard. That's the moment most of the funnel's value, and most of its risk, is concentrated. Get this fork right, and the rest of the line is downstream.
The exits

Six places where customers leave.

D1 · 20% Station 03 · Auth

Platform lead denies the GitHub App install

The marketer asks for Kite; the VP-Eng or platform team is the one approving the permission scope. If the install scope feels broad — especially the deploy-adjacent scopes — they say no on instinct. This is the load-bearing risk from the v0.4 research read.

Show only the minimum scope at install time. Reserve deploy / write scopes for the gold-line graduation step. Add inline plain-English explanations next to every permission.
D2 · 12% Station 04 · Scan

Stack Kite can't manage yet

The scan returns WordPress, Wix, Squarespace, Webflow, or a custom monorepo with exotic Webpack config. Kite can't credibly own the runtime, so the journey ends here.

Catch this at Station 01, not at 04 — pre-flight the URL the moment it's pasted. For unsupported stacks, capture the framework as a waitlist signal instead of leaving the user with a dead end.
D3 · 8% Gold · G2 · Credentials

Credentials withheld by their security team

Replicating the codebase needs API keys, env vars, CMS tokens, CDN credentials. The platform team approved the GitHub App install but draws a hard line at handing over secrets — especially production ones.

Two paths: (a) Start with staging-only credentials and gate prod-creds behind explicit migration. (b) Offer a Vault-style integration so secrets stay in the customer's vault and Kite reads them at runtime, not at install.
D4 · 7% Gold · G3 · Replicate

"Extras" Kite can't replicate

The site depends on things outside the repo: a Contentful pipeline, a Segment proxy, an internal auth gateway, a custom CDN ruleset. Kite spins up the runtime and pieces are missing. The "works on Kite, breaks in prod" failure mode the v0.4 thesis was meant to prevent — reintroduced through a different door.

Run a pre-flight "extras inventory" at Station 04 (scan), not at G3. Tell the customer up-front what we can replicate, what stays connected via their accounts, and what we can't take. Set expectations before we charge.
D5 · 5% Blue · B2 · PR

Engineering team rejects the PR

Kite's diff doesn't match the team's house style — wrong import ordering, wrong component pattern, touched files outside scope. PR sits, then closes. The blue-line user loses the moment Kite was supposed to make easy.

Run a "style match" pass against the repo's recent merged commits before opening any PR. Match conventions; don't impose them. Add an auto-message on the PR explaining what Kite did and what it deliberately didn't touch.
D6 · 6% Blue · B3 · Wait

Customer's CI/CD is too slow

The PR gets merged, but their pipeline takes 14 minutes to build and deploy. The snap-speed feeling never arrives. From the user's seat, it still looks like a Kite failure. The exact reason the gold line exists.

Surface the expected wait time inside Kite the moment the PR merges — borrowed from GitHub Actions / Vercel UI. And use this as the upsell trigger to graduate to the gold line: "Want this in 12 seconds instead of 14 minutes? Move to Kite-owned deploy."
